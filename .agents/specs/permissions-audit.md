# Permissions Audit Spec

Use this spec when asked to test Santuario Maringa user permissions end to end.

## Goal

Verify every app-level permission from `apps/mobile/src/lib/permissions.ts` against:

- API authorization behavior.
- Simulator UI visibility/navigation behavior.
- Permission propagation after changing a user's role.

At the end, report fixtures created, mutations performed, cleanup performed, and pass/fail results per permission.

## Permission Sources

App-level permissions are defined in `apps/mobile/src/lib/permissions.ts`:

- `users.read`
- `users.create`
- `users.update`
- `users.delete`
- `roles.read`
- `roles.create`
- `roles.update`
- `roles.delete`
- `cats.manage`
- `adoptions.manage`
- `adoption_candidates.manage`

Default role/permission data and app-level route enforcement live in `apps/api/src/routes/admin.ts`.

Important distinction: user create/list/update actions in the mobile app call Better Auth admin endpoints under `/auth/admin/*`, while roles and app-specific user delete use app routes under `/admin/*`.

## Before Testing

1. Read current code for:
   - `apps/api/src/routes/admin.ts`
   - `apps/api/src/routes/cat.ts`
   - `apps/api/src/routes/adoption-candidates.ts`
   - `apps/api/src/routes/adoptions.ts`
   - `apps/mobile/src/lib/permissions.ts`
   - `apps/mobile/src/components/app-tabs.tsx`
   - `apps/mobile/src/app/private/usuarios.tsx`
   - `apps/mobile/src/app/private/funcao/[id].tsx`

2. Confirm local API and simulator state:
   - API base is usually `http://localhost:8000`.
   - Mobile env is `apps/mobile/.env`.
   - API env is `apps/api/.env`.
   - Use `curl -i http://localhost:8000/me/permissions`; `401` means the API is alive but unauthenticated.

3. Use the existing admin cookie jar if available:
   - `cookies.txt` is in curl Netscape format.
   - Lines beginning with `#HttpOnly_` still contain real cookies; parse them by removing only the `#HttpOnly_` prefix.

4. For Better Auth admin endpoints, include:
   - `Origin: http://localhost:8000`
   - `Content-Type: application/json`

   Without the Origin header, Better Auth may return `MISSING_OR_NULL_ORIGIN`, which is not a permission result.

## Fixture Strategy

Create one dedicated test user and one dedicated custom app role:

- User email pattern: `codex-permission-tester-YYYYMMDD@example.com`
- Username pattern: `codex_perm_YYYYMMDD`
- Role name pattern: `Codex Permission Audit YYYYMMDD`
- Role slug is generated by the API, usually `codex-permission-audit-YYYYMMDD`.

Recommended final state after testing:

- Keep the test user and role if useful for follow-up simulator checks.
- Restore the role's `permissionKeys` to `[]`.
- Remove all disposable users, roles, cats, candidates, and adoptions created during probes.

Do not delete real users or existing data. Only delete objects with clear `Codex ...` names or IDs captured during the test run.

## API Test Matrix

For each permission:

1. Patch the test role to `[]`.
2. Sign in as the test user.
3. Confirm `/me/permissions` does not include the permission.
4. Call the relevant endpoint and expect `403`.
5. Patch the test role to `[permission]`.
6. Sign in again as the test user.
7. Confirm `/me/permissions` includes the permission.
8. Call the relevant endpoint and expect a 2xx success.

Endpoints to test:

- `users.read`: `GET /auth/admin/list-users?limit=5`
- `users.create`: `POST /auth/admin/create-user`
- `users.update`: `POST /auth/admin/update-user`
- `users.delete`: `DELETE /admin/users/:id`
- `roles.read`: `GET /admin/roles`
- `roles.create`: `POST /admin/roles`
- `roles.update`: `PATCH /admin/roles/:id`
- `roles.delete`: `DELETE /admin/roles/:id`
- `cats.manage`: `POST /cat`
- `adoption_candidates.manage`: `POST /adoption_candidates`
- `adoptions.manage`: `POST /adoptions`

Known risk to re-check: Better Auth admin endpoints may ignore custom app-role permissions unless the role also exists in Better Auth's static access control config.

Known risk to re-check: operational routes may only require a session and may not enforce `cats.manage`, `adoptions.manage`, or `adoption_candidates.manage` server-side.

## Simulator UI Checks

Use Computer Use to inspect the simulator.

Minimum checks:

- Login as admin and verify the user menu includes `Usuarios` when the user has user/role read permissions.
- Login as the test user with `[]` permissions.
- Confirm privileged tabs/actions disappear:
  - `Gatos`
  - central create action
  - `Candidatos`
  - `Adocoes`
  - user menu `Usuarios`
- Grant one visible permission at a time and check expected UI:
  - `cats.manage` shows `Gatos` and contributes to central create action.
  - `adoption_candidates.manage` shows `Candidatos` and contributes to central create action.
  - `adoptions.manage` shows `Adocoes` and contributes to central create action.
  - `users.read` or `roles.read` shows user/admin access.
  - `users.create`, `cats.manage`, `adoption_candidates.manage`, or `adoptions.manage` enables `/private/cadastrar`.

The current permissions query has `staleTime: 60_000`, so permission changes may not appear immediately. Prefer logout/login for clean UI checks. If a fresh login still does not reflect changed permissions, report it explicitly as a propagation/UI-refresh failure.

If iOS shows `Save Password?`, tap `Not Now`. Do not save test credentials.

## Cleanup

Cleanup order matters:

1. Delete temporary adoption rows first.
2. Delete temporary cats.
3. Delete temporary adoption candidates.
4. Delete disposable auth users.
5. Delete disposable app roles.
6. Restore retained test role permissions to `[]`.

Adoptions restrict deletion of linked cats/candidates, so deleting cats/candidates first may fail.

For direct cleanup through Drizzle, run from `apps/api` so API-local dependencies resolve:

```sh
bun - <<'EOF'
import './src/env.ts';
import { db } from './src/db/index.ts';
import { adoption } from './src/db/schema/adoption.ts';
import { cat } from './src/db/schema/cat.ts';
import { adoptionCandidate } from './src/db/schema/adoption-candidate.ts';
import { like } from 'drizzle-orm';

await db.delete(adoption).where(like(adoption.observation, 'permission audit%'));
await db.delete(cat).where(like(cat.name, 'Codex Cat%'));
await db.delete(adoptionCandidate).where(like(adoptionCandidate.name, 'Codex Candidate%'));
EOF
```

Only use direct DB cleanup for objects created by the audit.

## Final Report Format

Include:

- Test user and role created.
- Final state of test role permissions.
- Which fixtures were cleaned up and which intentionally remain.
- API pass/fail summary by permission.
- Simulator UI pass/fail summary.
- Any mismatches between UI gating and server-side enforcement.
- Location of any raw result file, if generated.

Keep the summary direct and call out security-relevant failures first.
